## 🪵 로그 상세 내역

### 2026-04-12 | Streamlit 환경에서 Runtime instance already exists! 오류 해결
- **배경 및 문제 정의 (가설):** `uv run streamlit run main.py` 명령어로 앱을 직접 실행할 때 `RuntimeError: Runtime instance already exists!` 오류가 발생하여 실행이 중단됨. Streamlit CLI 도구(`streamlit run`)로 실행 중인데 `main.py` 파일 내부에서 `stcli.main()`을 또다시 호출하여 런타임 객체를 중복 생성하려 한 것이 원인임을 파악함.
- **시도 및 해결책:** `streamlit.runtime.exists()` 메서드를 가져와서 런타임 존재 여부를 먼저 확인하도록 로직을 수정함 (`if not runtime.exists():` 조건문 추가). 
- **결과 및 채택 이유:** Streamlit 명령어로 실행했을 때는 내부에서 다시 재실행을 방지하여 에러가 깔끔하게 해결됨. 추후 사용자가 단순히 `python main.py`(또는 `uv run python main.py`)명령어를 입력했을 때도 `streamlit` 커맨드로 전환해주는 편의성은 그대로 유지할 수 있음.

### 2026-04-12 | 최신 무료 LLM 모델(Gemini 2.5 Flash)로 일괄 마이그레이션
- **배경 및 문제 정의 (가설):** 기존 `gemini-2.0-flash` 모델이 Deprecated(감손) 상태임을 공식 문서(`gemini-ai-studio.md`)에서 확인. 비용 효율적이면서도 동일한 성능을 보장하는 무료 티어 모델로의 전환 필요성 대두.
- **시도 및 해결책:** 프로젝트 로직 전반(`main.py`의 `ChatGoogleGenerativeAI`, `create_agent`)과 프로젝트 가이드라인(`AGENTS.md`)을 최신 권장 무료 모델인 `gemini-2.5-flash`로 모두 업데이트함. 포트폴리오 가이드에도 명시.
- **결과 및 채택 이유:** 서비스 안정성 확보 및 API 지원 중단에 대비 완료. 비용 발생 없이 지속적인 테스트와 RAG 고도화 작업을 이어나갈 수 있는 기반을 다짐.

### 2026-04-18 | 교육용 다층 가드레일 모듈(`src/guardrails.py`) 신설
- **배경 및 문제 정의 (가설):** 챗봇이 단순히 지식을 전달하는 것을 넘어, 부정행위 방지, 개인정보 보호, 심리적 위기 감지 등 교육 서비스로서의 최소한의 안전 장치(Safety Layer)가 부재함. `01_guardrails.ipynb`에서 검증된 4단계 가드레일 로직을 실제 서비스 모듈로 통합하여 시스템의 안정성과 교육적 가치를 높이고자 함.
- **시도 및 해결책:** `src/guardrails.py` 모듈을 신설하고 LangChain의 `before_agent`, `after_agent` 미들웨어 데코레이터를 활용하여 4층 방어막을 구축함. (1) 교육 집중도 유지, (2) PII 마스킹, (3) 상담 이관, (4) 정답 유출 방지 기능을 순차적으로 구현함. 특히 출력 단계에서는 `gemini-2.5-flash-lite` 모델을 활용해 답변의 적절성을 실시간 검증하도록 설정함.
- **결과 및 채택 이유:** 가드레일 로직을 `main.py`에 직접 넣지 않고 독립된 모듈로 분리함으로써 코드의 가독성과 유지보수성을 높임. `gemini-2.5-flash-lite` 사용으로 검사 레이어의 비용 발생을 최소화하면서도 모델 기반의 정교한 필터링이 가능해짐.

### 2026-04-18 | 메인 애플리케이션에 가드레일 시스템 통합
- **배경 및 문제 정의 (가설):** 신설된 `src/guardrails.py` 모듈이 실제 `main.py`의 에이전트 실행 흐름에 연결되지 않으면 보호 기능이 작동하지 않음. 실시간 대화 흐름에서 입출력 필터링이 즉각적으로 적용되도록 에이전트 초기화 로직을 수정해야 함.
- **시도 및 해결책:** `main.py` 상단에 가드레일 미들웨어 4종을 임포트하고, `initialize_agent` 함수 내 `create_agent` 호출 시 `middleware` 파라미터에 해당 함수들을 리스트 형태로 등록함.
- **결과 및 채택 이유:** 이제 사용자가 질문을 입력할 때마다 교육적 의도 확인, 개인정보 마스킹, 위기 상황 감지가 자동으로 수행되며, 답변 생성 후에는 정답 유출 여부를 한 번 더 검증함. 분리된 모듈 구조 덕분에 `main.py`의 코드는 깔끔하게 유지되면서도 강력한 보안/교육 정책을 적용할 수 있게 됨.

### 2026-04-18 | 가드레일 대응 Streamlit UI 최적화
- **배경 및 문제 정의 (가설):** 가드레일에 의해 차단되거나 수정된 답변은 일반적인 AI 답변과 성격이 다름(경고, 안내, 상담 이관 등). 이를 일반 텍스트로만 노출하면 사용자가 상황의 심각성이나 의도를 명확히 인지하기 어려울 수 있음.
- **시도 및 해결책:** 답변 핸들링 로직에 조건문을 추가하여 특정 이모지(`🚫`, `⚠️`, `⏰`)나 키워드(`(상담실 연결 중...)`)를 감지하도록 함. 감지된 유형에 따라 `st.error`, `st.warning`, `st.info` 등 Streamlit의 시각적 컴포넌트를 사용하여 답변을 차별화하여 출력함.
- **결과 및 채택 이유:** 단순 텍스트보다 시각적으로 강조된 메시지를 제공함으로써 교육적 피드백의 효과를 높이고, 상담 이관과 같은 중요 상황에서 사용자의 주의를 효과적으로 환기할 수 있게 됨.

### 2026-04-18 | 통합 가드레일 시스템 성능 테스트 및 키워드 최적화
- **배경 및 문제 정의 (가설):** 구현된 가드레일 레이어가 실제 다양한 사용자 입력 시나리오(부정행위, 딴짓, 개인정보 입력, 위험 상황)에서 의도한 대로 작동하는지 검증이 필요함. 초기 테스트 결과 '리그오브레전드'와 같은 구체적인 게임명이 필터링되지 않는 문제 발견.
- **시도 및 해결책:** 별도의 테스트 스크립트(`scratch/test_guardrails.py`)를 작성하여 5가지 시나리오를 자동 테스트함. 테스트 과정에서 누락된 '리그오브레전드', '발로란트' 등의 키워드를 `FORBIDDEN_TOPICS`에 추가하고, 이메일/전화번호 마스킹 시 AI가 실제 원본 데이터를 보지 못하고 `<REDACTED>` 형태로 수신함을 로그를 통해 확인함.
- **결과 및 채택 이유:** 테스트 결과 모든 레이어가 정상 작동함을 확인(부정행위 100% 차단, PII 완전 익명화, 위기 시 상담 이관 로직 발동). 특히 키워드 기반 필터링의 한계를 보완하기 위해 주요 게임 및 유해 키워드를 확장하여 방어력을 높임.
